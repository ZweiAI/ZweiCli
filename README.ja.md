## なぜか

ほとんどのコーディングエージェントは、すべてを 1 つのコンテキストに詰め込みます：読み、書き、テストを実行し、採点し、リトライする。実行が長くなるほど注意力は薄まり、テスト出力が意図をかき消し、最終的にエージェントは自分の宿題を自分で採点する羽目になります。

Zwei はアカデミアのパターンを借用しています：**PhD が書き、Supervisor が採点する。** 2 つの隔離されたセッション、2 つの独立したスキルセット、境界では情報が一方向にのみ流れる。書き手は採点者の推論を覗き見ることができない — だからそれに合わせて最適化することもできません。

## 核心となる賭け

Zwei の前提はシンプルです：1 つのコーディングエージェントを**隔離された 2 つの役割**に分ける — PhD が書き、Supervisor がレビューする — そして情報は**一方向**にのみ流れる。書き手は採点者の推論を決して見ません。

これは実在する失敗モードを狙っています：**単一エージェントの自己評価は長い会話の中で劣化していく**（いわゆる「Goodharting」 — 自分の宿題を自分で採点し、いつでも A を付けてしまう現象）。

具体例を挙げましょう。エージェントにソート関数を書かせ、自己テストさせるとします：

- **シングルエージェントモード：** バグのあるテストを書き、その自作のバグ入りテストを「パス」して、勝利宣言する。
- **Zwei のデュアルエージェントモード：** PhD は関数を書きますが、**テストは見えず、`bash` 権限もありません**。Supervisor が独立してコードをレビューし、テストを実行します。両者のコンテキストが互いを汚染することはありません。

これは**アーキテクチャ上の賭け**です：_役割の隔離 + 情報の非対称性 > 単一コンテキストですべてをこなす_。この方向性は探求する価値があります — 特に複雑で多段階のコーディング作業において。しかし現時点で Zwei は**検証済みのツールというより、有望な実験**として理解するのが適切です。エージェントアーキテクチャに関心があるなら一見の価値あり。日常的に使える高効率なコーディングアシスタントを探しているだけなら、もう少し待ってから戻ってきてください。

## 核心となる考え方

- **デュアル・アテンション** — 2 つのセッションが物理的に隔離されます。PhD は執筆に集中し、Supervisor はレビューに集中します。どちらの役割も相手の仕事で注意力を消費しません。
- **独立したスキル** — 各役割は自分専用のスキルセットをロードします。デフォルトでは何も共有されません — PhD はレビューツールに気を取られず、Supervisor も手を伸ばして編集したくなる誘惑がありません。
- **非対称なメモリ** — PhD は Supervisor の推論を決して見られず、境界を越えるのは構造化された判定のみです。Supervisor は PhD のコードとテスト出力を見られます。メモリは設計上、一方向にのみ流れます。
- **執筆 / テストの分離** — PhD は `bash` を持たず、テストファイルへの読み取りアクセスもありません。自己検証は物理的に不可能であり、書き手のコンテキストはクリーンに保たれます — ツールの痕跡、テスト出力、ファイルダンプによる汚染はありません。

合わせた効果：書き手は Supervisor の言い回しを推し量って検証器を Goodhart することはできず、両者のコンテキストも互いの産物で汚染されません。

## インストール

### ソースから

```bash
git clone https://github.com/ZweiAI/ZweiCli
cd ZweiCli
bun install
bun run --cwd packages/zwei dev --help
```

### npm から

```bash
npm install -g @zweicli/cli
zwei --help
```

自動更新はデフォルトで有効で、npm 上の `@zweicli/cli@latest` を追跡します。手動で更新するには：`zwei upgrade`、または `npm install -g @zweicli/cli@latest`。

デュアルループ以外のすべて（認証、モデル、プロバイダー、セッション、Web UI）については、上流 opencode の規約がそのまま適用されます。[opencode.ai](https://opencode.ai) を参照してください。

## 使い方

TUI を起動：

```bash
zwei
```

### スラッシュコマンド

TUI 内でプロンプトに `/` を入力します。デュアルエージェントのワークフローに関連するコマンド：

| コマンド | 機能 |
|---|---|
| `/agents`（または `/agent`）| モード × ロールを選択。下のモード表を参照 |
| `/model` | **PhD と Supervisor の両方**のモデルを変更 |
| `/model1` | **PhD のみ**（書き手）のモデルを変更 |
| `/model2` | **Supervisor のみ**（採点者）のモデルを変更 |
| `/clear` | 3 つのセッションすべて（あなた + PhD + Supervisor）の会話をクリア |
| `/clear1` | PhD のセッションのみクリア |
| `/clear2` | Supervisor のセッションのみクリア |

### `/agents` — モードを選ぶ

agents ダイアログには 6 つのオプションが表示されます — 3 つの**モード** × 2 つの**ロール**の組み合わせです：

| モード | 動作 | 使いどころ |
|---|---|---|
| **`dual`** | 毎ラウンド、PhD が書いた後に Supervisor がレビュー。Supervisor は**常に動く** | 長いタスク、厳格なレビュー、Goodhart 対策の評価設定 |
| **`auto`** | PhD が先に書く。テストゲートを通れば Supervisor はスキップ、そうでなければ呼び出される | デフォルト — 書き手が 1 ラウンド目で決めればトークン節約 |
| **`single`** | PhD のみ、Supervisor なし。上流 opencode のシングルエージェントフローと同等 | 強力なモデルがワンショットで解決できるタスク — レビューに金を払う意味がない |

**ロール**接尾辞（`fast` vs `plan`）はエージェントのバリアントを決定します：

- **`fast`** — 実行モード；エージェントが実際に編集と実行を行う
- **`plan`** — プランニングモード；読み取り専用、fast に切り替える前にプランドキュメントを作成

つまり `dual fast` は「PhD + Supervisor、両方とも実行モード」、`auto plan` は「PhD がまず計画、Supervisor が要求に応じて計画をチェック」、といった具合です。

### PhD と Supervisor に異なるモデルを使う

非対称分割の肝は、書き手を安価に、採点者を強力に（あるいはその逆）できることです：

```
/model1   # PhD に速くて安いモデルを選択（例：Haiku 4.5）
/model2   # Supervisor に強くて厳しいモデルを選択（例：Opus 4.6）
```

実行途中でのモデル切り替えも可能です — 変更は現在のラウンドではなく、次のラウンドから反映されます。
